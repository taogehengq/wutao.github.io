RAG

什么是RAG?

RAG（Retrieval-Augmented Generation，检索增强生成）
一种将“信息检索”与“生成式大模型”耦合的架构。系统先通过稠密向量或稀疏检索，从大规模外部知识库中定位与当前查询最相关的文档片段（context），再把这些片段连同原始查询一起送入大语言模型（LLM）作为 prompt 的上下文，让模型在生成回答时既利用内部参数知识，又利用最新的外部知识，从而显著降低幻觉、提升时效性与可溯源性。其典型流程：Indexing → Retrieval → Augmentation → Generation。一句话解释就是：先用“搜索引擎”找到相关文档，再让“作文高手”参考这些资料写答案，既新鲜又靠谱。

概览

  • 建议走“螺旋式”路径，而不是完全线性的“原理→框架→二开→自研”。先跑通业务、立评测，再围绕问题点补原理、做定制，最后在关键环节做组件级自研。
  • 学习目标不是会调API，而是能“量化提升质量、解释问题、降本提效”。每个阶段都要有可复现的指标与产出。

一、总体路线（6–8周，可压缩/拉伸）
阶段0｜准备（0.5周）

  • 预备知识：Python、基础IR（TF-IDF/BM25）、向量与余弦相似度、基本Linux与容器。
  • 数据：选一个你熟悉且可开源展示的小型领域集（公司规章/开源项目文档/金融研报/中文法规），准备10–30个“黄金问答”作为评测集。

阶段1｜先跑通（1–3天）

  • 用一个框架快速搭端到端 RAG：LlamaIndex/Haystack/RAGFlow/LangChain 任选其一。
  • 最小可行：本地文档→切分→嵌入→向量库→检索→重排→回答→引用。
  • 交付物：可用Demo + 基线指标（例如：命中率/引用正确率/平均延迟/每次调用成本）。

阶段2｜评测先行（2–3天）

  • 建黄金集（问题、标准答案、证据文档），用 Ragas/DeepEval/TruLens 做离线评测。
  • 建立迭代仪表：召回@k、nDCG、答案相关性、引用正确率、延迟、成本。
  • 交付物：基线评测报告 + 可复现实验脚本。

阶段3｜原理补齐 + 针对性优化（1–2周）

  • 学什么原理？只学和你当前瓶颈相关的：
    • 召回不稳：混合检索（BM25+向量）、多路召回（multi-query/RAG-Fusion）、HyDE。
    • 回答偏题/幻觉：重排（bge-reranker/ColBERT）、强制引用、提示结构化。
    • 找不到信息：切分策略（标题感知、层级切分、滑窗/重组）、元数据过滤。
    • 延迟高/成本高：ANN索引参数（HNSW/IVF）、缓存、批量化、流式生成。
  • 轻度“二次开发”：替换组件、改切分器、加多路检索与重排。
  • 交付物：对比报告（例如召回@20 从0.62→0.78、引用正确率从0.70→0.86、P95延迟-25%），并配误差分析。

阶段4｜组件级自研（1–2周，选一个深入）

  • 选一个点做“能讲得清、拿得出数据”的深挖：
    • 自研切分/重组：段落+标题+表格/代码结构化，支持章节回填。
    • 实现一个“最小RAG”纯代码版（约100–200行）：Sentence-Transformers + FAISS + 自写BM25 + bge-reranker。
    • 检索策略研究：RAG-Fusion/回溯重排/多跳检索；或自训练一个小型Reranker。
  • 交付物：技术笔记/博客 + 代码仓库 + 指标提升曲线。

阶段5｜工程化意识（1周）

  • 增量索引/版本管理、缓存（结果/嵌入/提示）、日志与观测、A/B测试。
  • 交付物：一页架构图 + 运维与成本优化清单。

二、学习顺序该如何权衡？

  • 不建议：先把所有原理学完再动手。你会在细节里迷路，且难以形成“问题→原理→验证”的闭环。
  • 推荐比例：做中学（60%）+ 原理补齐（30%）+ 从零实现/深挖（10%）。
  • 进阶阈值（什么时候从“二次开发”跨到“自研组件”）：
    • 你的评测卡在某个瓶颈（比如召回或引用正确率）连续两周难以提升；
    • 业务提出刚性目标（延迟、成本、合规、权限）现有框架难满足；
    • 你能明确“自研该组件的收益>两周投入”。

三、能力里程碑（自检清单）

  • L1 可用：能用任一框架在你的数据上搭出稳定RAG，含引用与混合检索。
  • L2 可评：会构建黄金集、用Ragas/DeepEval跑自动评测，会做误差分类与定位。
  • L3 可优：能通过切分/混合检索/重排/查询改写把关键指标提升20–30%。
  • L4 可讲：能写清楚方法、实验设置与指标，给出可复现实验仓库。
  • L5 可造：至少在一个环节实现可替换组件（切分器/检索器/重排器/评测管线）。

四、推荐技术栈（可替换）

  • 模型
    • Embedding: bge-m3 或 gte-Qwen2；中文可先用 BAAI/bge-small-zh。
    • Reranker: bge-reranker-v2-m3 或 Cohere Rerank（云端）。
    • LLM: Qwen2.5/GLM-4/Claude/ GPT-4o 或本地Qwen2.5-7B-Instruct。
  • 检索与索引
    • 向量库：pgvector（轻量）、Qdrant/Weaviate/Milvus（规模化）。
    • 关键词：Elasticsearch/OpenSearch 的 BM25；做混合检索。
  • 框架与工具
    • LlamaIndex / Haystack / LangChain / RAGFlow(快原型)；
    • 评测：Ragas/DeepEval/TruLens；可配 Promptfoo 做对比。
  • 实用技巧
    • 切分：标题感知 + 适度重叠（如256–512 tokens，overlap 15–20%）；
    • 检索：混合检索 + Rerank top-50→top-10；
    • 查询改写：multi-query 或 RAG-Fusion；
    • 强化引用：要求模型以“证据片段+页码/段落ID”回答。

五、练手项目建议（能进作品集）

  • 法规/公司制度助理：强调有效期与引用正确率；做“过期文档过滤”。
  • 工程文档助理：对代码/表格友好切分；支持多跳问答（先定位API，再找示例）。
  • 多语检索：中英文混写资料库，验证 bge-m3 跨语种检索效果与Rerank收益。
    每个项目都要有：数据清单、评测集、指标提升表、误差分析和成本对比。

六、关键原理清单（按需补齐）

  • IR与ANN：BM25/TF-IDF、HNSW/IVF/量化、向量归一化与距离度量。
  • 检索策略：混合检索、多路召回、HyDE、RAG-Fusion、多跳检索。
  • 生成与控制：提示结构化、引用强约束、答案去重与合并。
  • 评测：召回@k、nDCG、MRR、引用正确率、事实一致性；人工校验方法与标注指南。
  • 工程：增量索引、版本化、缓存、速率限制、成本/延迟监控。

七、常见坑

  • 不做混合检索，只靠向量，导致召回不稳。
  • 切分粗糙，忽略标题/层级/表格/代码结构。
  • 无黄金集与自动评测，只靠主观“感觉还行”。
  • 只追新模型，不做误差分析与问题定位。
  • 指标不成体系：离线提升与线上体验不一致。

八、如果你想要一条“极简开始”的清单

  • Day 1–2：用 LlamaIndex + pgvector + bge-m3 跑通，出基线。
  • Day 3–4：补黄金集 + Ragas，出第一版评测。
  • Week 2：加 BM25 混合检索 + bge-reranker；改切分；写提升报告。
  • Week 3：实现最小RAG（纯Python+FAISS）；对比框架效果与延迟/成本。
  • Week 4：择一深挖点（RAG-Fusion 或结构化切分），写技术文章。

最后的建议

  • 你的顺序可以是“先框架跑通→立评测→围绕指标补原理→针对性二开→选择一处自研”。这比“先把所有原理学完再动”更高效。
  • 如果愿意，告诉我你的背景（工程/算法/产品）、可投入时间、目标行业与数据规模，我给你定制一份更细的周计划和工具选型。
